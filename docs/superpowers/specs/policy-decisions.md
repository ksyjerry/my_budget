# Policy Decisions Tracker

**시작일**: 2026-04-25
**참고**: [feedback-0425 메타 프레임워크 1.4](2026-04-25-feedback-0425-systematic-framework-design.md#14-정책-결정-트랙-병렬)

## 운영 규칙

POL 결정 완료의 정의 (3가지 모두 충족):
1. 결정자가 명확한 답변 제공 (구두만으로 부족, 메일/Teams 등 텍스트 형태로 확보)
2. 본 문서에 결정 사항 + 결정자 + 결정 일자 + 출처(메일·회의록 등) 기록
3. 본 시스템 owner가 기록을 확인·승인

3가지 중 하나라도 빠지면 미결정 상태로 간주, 영역 진입 차단.

상태 값:
- `미결정` — 결정자에게 요청 전 또는 답변 대기 중
- `답변 수신` — 결정자 답변 받았으나 owner 확인 전
- `결정 완료` — 3가지 조건 모두 충족
- `재논의` — 답변 받았으나 후속 영역 진행 중 충돌 발견하여 재요청

## POL-04: EL 승인/취소 워크플로우

| 항목 | 값 |
|---|---|
| 결정자 | 김동환 |
| 차단 영역 | 2, 3 |
| 상태 | **답변 수신 (provisional — owner 잠정 승인)** |
| 결정값 | **(b) 표준형 — 작성중 → 작성완료(PM) → 승인(EL) 3단계 단방향. 락 상태에서 수정하려면 EL이 락 해제 후 재편집** |
| 결정 일자 | 2026-04-25 |
| 출처 | 사용자(owner) 잠정 승인 — 외부 결정자(김동환) 컨펌 후 정식 확정 예정. 추천 근거: 피드백 #61이 명시한 "EL 승인 / 취소 / 변경" 모두 충족, 일반 감사 워크플로우 패턴, YAGNI 준수 |
| owner 확인 | ✅ 사용자 잠정 승인 (2026-04-25) — 김동환 텍스트 컨펌 받기 전까지 provisional |

**Provisional 의미:**
- 본 결정으로 영역 2 진입 가능. 영역 2/3 코드 작업이 (b)를 전제로 진행됨
- 외부 결정자(김동환) 컨펌이 다른 안으로 결정되면 영역 2/3 종료 후 재작업 필요할 수 있음
- 이는 메타 spec 1.7 "명시적 트레이드오프 정책"의 (a) 일정 연기 vs (b) 영역 범위 축소 vs (c) 안전망 일부 생략과는 다른 — 사용자(owner)가 자기 권한 내에서 잠정 결정한 case
- 외부 결정자 컨펌이 (a)/(c)로 오면 별도 fix 라운드 필요

**배경**:
- 피드백 #61: "작성완료된 프로젝트의 Time budget을 변경하고 싶은데 변경버튼이 없습니다. 변경이 가능했으면 좋겠습니다. 취소 기능 및 EL 승인 기능도 있었으면 좋겠습니다."
- 피드백 #98: "budget 자료를 수정하기 위해서는 수정버튼이 따로 있으면 좋을 것 같습니다. 지금은 수정이 바로 가능합니다."

**핵심 결정 필요 사항**:
- (a) 워크플로우 상태 정의: 작성중 → 작성완료 → 승인대기 → 승인완료 → (취소 가능?) → 수정요청
- (b) 권한 분리: 작성자(PM) / 승인자(EL) / 관리자
- (c) 작성완료 후 수정 시 어떤 상태로 돌아가는가 (작성중? 승인대기?)
- (d) 승인 취소 권한자 (EL? 관리자?)
- (e) 작성완료 → 작성완료 → 작성완료 같은 무한 수정 허용 여부
- (f) 수정 이력 기록 범위 (현재 budget_change_log 존재 — 활용 또는 확장)

**선택지 (개략)**:
- 단순형: 작성중 ↔ 작성완료 (현행에 수정 버튼만 추가, EL 승인 없음)
- 표준형: 작성중 → 작성완료(PM) → 승인(EL) (3단계, 단방향)
- 확장형: 표준형 + 취소·수정요청·재제출 (양방향)

**영향**:
- 영역 2 페이즈 A: 목록 화면에 표시할 상태 enum 결정
- 영역 3 페이즈 A: Step 1 수정 모드(#98) 진입 조건
- DB 스키마: project.status 또는 budget_workflow_state 테이블
